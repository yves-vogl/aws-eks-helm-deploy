## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `boto3.Session` credentials → EKS describe_cluster API call | The SigV4-signed `describe_cluster` request authenticates to AWS; the response carries the cluster endpoint + CA (public material). No credential material flows back. |
| `ClusterAccess.endpoint` + `ca_data` + `token` → kubeconfig tempfile on local disk | The token is a secret (15-min EKS bearer). The endpoint + ca_data are public. The kubeconfig file is the consolidation point of the secret + non-secrets. `chmod 0600` before write closes the race window. |

| `ClusterAccessError` / `KubeconfigError` → `pipe.fail()` | Error messages contain public AWS error codes + paths. No credential material in the message text. |

## Documented Deviations from CONTEXT / RESEARCH

### Deviation 1: `KubeconfigError.exit_code = 7` (NOT 5 as CONTEXT D8 cites)

**CONTEXT D8 claim:** the error mapping table lists `KubeconfigError` at `exit_code = 5`.

**Reality (per RESEARCH Section L "Error Hierarchy Reconciliation"):** Exit code 5 is already taken by `HelmError` (Phase 1 placeholder). CONTEXT D8's table was written without cross-referencing the existing `errors.py`. Three errors collide:
- `KubeconfigError` (D8 says 5) — collides with `HelmError`
- `HelmExecutionError` (D8 says 6) — collides with `HelmTimeoutError`
- `HelmTimeoutError` (D8 says 7) — exists at 6

**Plan response:** Preserve existing exit codes 1..6 unchanged (RESEARCH Section L Recommendation 1). Add new errors with non-colliding integers:
- `KubeconfigError = 7` (this plan)
- `HelmExecutionError = 5` (Plan 03-02 — RENAME of existing `HelmError`; backward-compat alias `HelmError = HelmExecutionError` retained)
- `HelmTimeoutError = 6` (no change — already exists)

**Rationale:** Renumbering existing codes is a breaking change for any consumer that grew a habit of catching specific exit codes. Phase 3 keeps the 1..6 contract stable; new errors take the next free integer.

**Action item:** Plan-Checker MUST verify Plan 03-02 follows the same convention (HelmError → HelmExecutionError rename + alias; new errors take 7+ only). CONTEXT D8 may be updated retroactively in Phase 7 documentation work; this plan does not block on that.

### Deviation 2: `write_kubeconfig` does NOT raise `KubeconfigError` directly

**CONTEXT D3 implication:** the kubeconfig writer is the "natural" place to raise `KubeconfigError`.

**Plan response:** the WRITER stays as a primitive — raw `OSError` / `PermissionError` propagate from `tempfile.NamedTemporaryFile`, `os.chmod`, and `path.write_text`. The ACTION layer (Plan 03-04's `actions/upgrade.py`) wraps the `with write_kubeconfig(...)` call in `try: ... except OSError as exc: raise KubeconfigError(...) from exc`.

**Rationale:** Two benefits:
1. **Testability** — `test_kubeconfig.py` can assert on raw `PermissionError` from a contrived `os.chmod` failure without coupling to `KubeconfigError`'s constructor signature.
2. **Error context** — at the action layer, the call site knows WHICH cluster, WHICH release, WHICH step failed. The writer only knows "file write failed" without action-level context. Raising at the action layer gives consumers more actionable messages.

This is a layering choice, not a behavior deviation. `KubeconfigError` ships in this plan (Task 03-1-01) and is USED in Plan 03-04. Plan-Checker MUST confirm Plan 03-04 has the `try: ... except OSError: raise KubeconfigError` wrapper.

### Deviation 3: `_build_kubeconfig_yaml` is private (no public re-export)

**RESEARCH Section B exposes a single public function pattern** — the YAML builder is the body of `write_kubeconfig`.

**Plan response:** factor out `_build_kubeconfig_yaml(cluster, token) -> str` as a PRIVATE module-level helper (underscore prefix). NOT exposed in `__all__`. The function remains accessible for direct unit testing if Phase 4/5 want to (e.g., snapshot-test the YAML structure independently). Not in scope for Phase 3 testing — `test_kubeconfig.py` covers the YAML structure via `yaml.safe_load(path.read_text())` after `write_kubeconfig` runs.

**Rationale:** keeps the public surface minimal (one function — `write_kubeconfig`) while leaving an internal seam for future tests. No additional Phase 3 tests required.
