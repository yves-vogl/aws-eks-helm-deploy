## Documented Deviations from CONTEXT / RESEARCH

### Deviation 1: UpgradeAction.__init__ accepts `strategy: AuthStrategy | None = None` kwarg

**CONTEXT D1 implies:** UpgradeAction wires `select_strategy â†’ ...` internally.

**Plan response:** Add an optional `strategy` constructor kwarg so `cli.py` can pass the already-selected strategy (it called `select_strategy` once to bind `auth_strategy` to structlog context). If `strategy is None`, UpgradeAction calls `select_strategy(self._settings)` itself â€” preserves the self-contained behavior for direct unit testing.

**Rationale:** prevents `select_strategy` being called twice per pipe invocation (once in cli.py for the bind_safe_context call, once in UpgradeAction for the actual credential fetch). Double-selection is harmless (idempotent for static-keys; for OIDC in Phase 4, it would mean two OIDC token exchanges) but wasteful.

**Action item:** Plan-Checker MUST confirm Task 03-4-02's `UpgradeAction.__init__` signature matches and Task 03-4-03's cli.py passes `strategy=strategy`.

### Deviation 2: `os.environ` read INSIDE `build_bitbucket_set_args` (not via Settings)

**Phase 1 / Phase 2 convention:** "no `os.environ` outside `settings.py`".

**Plan response:** BITBUCKET_* vars are read directly via `os.environ.get(...)` per CONTEXT D5 + RESEARCH Section G. Documented exception with rationale â€” mirrors `auth/__init__.py::_derive_session_name` which reads BITBUCKET_PIPELINE_UUID + BITBUCKET_BUILD_NUMBER directly.

**Rationale:** BITBUCKET_* vars are PLATFORM-supplied (Bitbucket Pipelines injects them at runtime per pipe step) â€” NOT consumer-supplied via pipe.yml. Wiring them through Settings would require 5 declarative fields per CONTEXT D5; this is more boilerplate than the `os.environ.get` pattern.

**Action item:** the module docstring of `actions/upgrade.py` documents this exception explicitly so future reviewers don't flag it as a Phase 1 convention violation.

### Deviation 3: `# pragma: no cover` on the `cli.py` `else: raise ConfigurationError` branch

**CONTEXT D1 + REQUIREMENTS TOOL-06:** 100% line + branch coverage required, no `# pragma: no cover` without PR review.

**Plan response:** the defensive `else` branch in cli.py's action dispatch CANNOT be exercised in Phase 3 because `Settings.action: Literal["upgrade"]` rejects any other value at parse time. Two options (see Task 03-4-03 behavior):
- Option 1 (preferred): `# pragma: no cover` with inline comment "Defensive â€” branch reachable once Phase 5 widens Settings.action Literal".
- Option 2 (zero-pragma): use a dispatch dict so the `else` is dropped entirely; Phase 5 adds entries to the dict.

**Plan resolution:** EXECUTOR's discretion within these two options. Plan-Checker MUST confirm the chosen option meets the 100% gate.

**Rationale:** the unreachable defensive branch is a forward-compat scaffold; pragma is the standard way to mark intentional unreachable defensive code. The alternative (dispatch dict) is over-engineering for one entry.

### Deviation 4: `--history-max 0` is passed through to helm (per CONTEXT D4 + Pitfall 3)

**Pitfall 3:** `HISTORY_MAX=0` means UNLIMITED retention to helm (counterintuitive).

**Plan response:** the action layer does NOT translate or block 0; `settings.history_max == 0` flows to `HelmClient.upgrade_install(history_max=0)` which emits `--history-max 0` (Plan 03-02 `_build_argv` checks `history_max is not None`, not `is truthy`). Test `test_history_max_0_passes_0_to_helm_client` enforces. Phase 7 docs explain the unlimited semantics.

**Action item:** Phase 7 (DOC-02 variable reference) MUST document HISTORY_MAX=0 means unlimited. Carry forward to Phase 7 backlog.

### Deviation 5: META-02 + META-03 (Phase 5) are NOT implemented here

**CONTEXT D5:** Phase 3 implements META-01 (opt-in injection); META-02 (default flip â€” already done via `inject_bitbucket_metadata: bool = Field(default=False)` from Phase 1, preserved) + META-03 (deprecation warning when chart references `.Values.bitbucket.*` without `INJECT=true`) ship in Phase 5.

**Plan response:** Phase 3 ships ONLY META-01. No deprecation warning logic in this plan. `settings.inject_bitbucket_metadata` defaulting to `False` is preserved from Phase 1; flipping that default would BE the META-02 work, which is Phase 5. No code change required for META-02 â€” it ships free as a consequence of the default value already being False.

**Action item:** Phase 5 plan-checker confirms META-03 detection logic is added at that time. No carry-over for this plan.
