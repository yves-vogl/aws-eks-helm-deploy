### Deviation 4: stderr truncation uses char slicing (not byte slicing)

**RESEARCH Section E claims byte-length truncation:** `if len(stderr) > STDERR_MAX_BYTES`.

**Plan response:** the implementation checks `len(s.encode("utf-8")) > STDERR_MAX_BYTES` (byte length for the threshold check — conservative) BUT applies `s[-STDERR_MAX_BYTES:]` (char slicing for the truncation — simpler, no UTF-8 boundary issues).

**Rationale:** helm stderr is ASCII-dominant; char slicing approximates byte slicing closely enough. Byte-precise slicing would risk truncating in the middle of a multi-byte UTF-8 code point. The 32 KB cap is a DoS mitigation, not an exact length guarantee — the char-slice approximation is within the spirit of CONTEXT D2.
