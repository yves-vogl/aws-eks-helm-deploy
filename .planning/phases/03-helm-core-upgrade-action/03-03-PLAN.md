## Documented Deviations from CONTEXT / RESEARCH

### Deviation 1: `ResolvedChart` is a concrete dataclass (NOT a Protocol)

**RESEARCH Open Question 2:** whether to introduce a `ChartSource` Protocol in Phase 3 vs Phase 4.

**Plan response:** Keep `ResolvedChart` as a concrete frozen dataclass per RESEARCH recommendation + CONTEXT D6. Phase 4 refactors to a Protocol when `RepoChart` + `OciChart` arrive (CHART-02, CHART-03).

**Rationale:** YAGNI — a Protocol with one implementation is over-engineering. The dataclass is the simplest shape that satisfies Phase 3's needs and Phase 4 will refactor non-breakingly (existing `ResolvedChart` implements the Protocol structurally).

### Deviation 2: `resolve_local_chart` accepts a `repo_root` parameter (NOT in RESEARCH Section H signature)

**RESEARCH Section H signature:** `parse_chart_yaml(chart_path: pathlib.Path) -> tuple[str, str]`.

**Plan response:** the public function is `resolve_local_chart(chart_spec: str, repo_root: pathlib.Path | None = None) -> ResolvedChart` — adds:
- the `repo_root` parameter so the function is testable without `monkeypatch.chdir` (a common pytest brittleness pattern).
- the `chart_spec: str` parameter shape so the action layer can pass `settings.chart` directly (which is a string from env var, not a Path).
- the return type is `ResolvedChart` (full value object) not `tuple[str, str]` — the source_path needs to be returned so HelmClient knows the helm CLI's `<chart>` argument.

**Rationale:** RESEARCH Section H's signature is a helper sketch; the public API surface needs the dataclass return + the env-string accept + a testable cwd-independent path resolution. Plan-Checker MUST confirm Plan 03-04 calls `resolve_local_chart(settings.chart)` (single argument; no `repo_root` — default `pathlib.Path.cwd()` is fine for the Bitbucket Pipelines runtime).

### Deviation 3: Empty Chart.yaml raises `ChartResolutionError` (per behavior contract)

**RESEARCH Section H** doesn't explicitly handle the `data is None` case (empty file).

**Plan response:** add an explicit `if data is None: raise ChartResolutionError` check. PyYAML's `safe_load("")` returns `None` (not an empty dict); without this check, the subsequent `data.get(...)` would raise `AttributeError` — unhelpful to consumers.

**Rationale:** explicit error with clear message ("Chart.yaml at {path} is empty or malformed") is more debuggable than a generic AttributeError.

### Deviation 4: Non-mapping Chart.yaml (YAML list or scalar at top level) raises `ChartResolutionError`

**RESEARCH Section H** assumes well-formed mapping Chart.yaml.
