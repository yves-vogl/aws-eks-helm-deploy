    def kind_kubeconfig(kind_cluster: str) -> Iterator[pathlib.Path]:
        try:
            result = subprocess.run(
                ["kind", "get", "kubeconfig", "--name", kind_cluster],
                check=True, capture_output=True, text=True, timeout=60,
            )
        except subprocess.CalledProcessError as exc:
            pytest.skip(f"kind get kubeconfig failed: {exc.stderr}")
        with tempfile.NamedTemporaryFile(

## Documented Deviations from CONTEXT / RESEARCH

### Deviation 0: `HISTORY_MAX` unset integration test deliberately not added (plan-check Warning 2)

**CONTEXT D4 says:** "integration tier: 6 sequential upgrades with `HISTORY_MAX=5` → `helm history` shows exactly 5. With `HISTORY_MAX` unset → default-10 behaviour holds."

**Plan response:** The `HISTORY_MAX=5` case IS integration-tested (Test 2 — 6 upgrades, asserts count == 5). The unset → default-10 case is NOT added as a separate integration test because:

1. The unit-tier coverage proves the wire behavior: 03-02 snapshot tests assert `--history-max` is absent from argv when `history_max=None`; 03-04 unit test asserts `settings.history_max=None` flows through to `HelmClient.upgrade_install(history_max=None)`.
2. Proving "default 10 holds" against a real cluster requires 11 sequential upgrades (then assert count == 10). At ~5 sec per upgrade plus kind cluster overhead this adds ~1 min of test runtime for no novel coverage — the assertion is about Helm's own default behavior, not our code.
3. Helm's `--history-max` default-10 behavior is documented upstream and stable across the 3.x series. Phase 3 inherits it, doesn't redefine it.

**Action item:** None at execution. Plan-Checker accepted-deviation rationale. Phase 5 (when rollback ships) MAY add an HISTORY_MAX-vs-REVISION integration test that incidentally covers this gap.

### Deviation 1: kind cluster name reuses Phase 1's `test-pipe-integration` (NOT CONTEXT's suggested `kind-phase3`)

### Deviation 2: kind admin kubeconfig path (NOT the EKS-token path)

**CONTEXT D10 originally posed an open question:** "does kind's kube-apiserver accept the moto-presigned-URL EKS token?"

**Reality (per RESEARCH Section C definitive finding + corrections #7):** NO. kind has no `aws-iam-authenticator` webhook. The EKS-token path is unit-tested via `@mock_aws` in Phase 2 + Plan 03-01; the kind integration uses kind's own admin kubeconfig.

### Deviation 3: Test 2 asserts EXACTLY 5 revisions (NOT ≤ 5)

**CONTEXT D4 says:** "6 sequential upgrades with `HISTORY_MAX=5` → `helm history` returns 5 revisions."

**RESEARCH Section F clarification:** helm prunes after each successful upgrade once the cap is reached. After 6 successful upgrades with cap=5, exactly 5 revisions remain (revisions 2-6 visible; revision 1 pruned).

**Plan response:** assert `len(revisions) == 5` (exact equality, not `<= 5`). This catches BOTH over-retention (cap not enforced) AND under-retention (over-aggressive pruning). Test docstring documents the exact-equality choice.

**Rationale:** stricter assertion + more diagnostic value if the test fails.

### Deviation 4: PipeIO is mocked with MagicMock(spec=PipeIO) (NOT the real bitbucket-pipes-toolkit Pipe)

**Phase 1 / 2 integration tests** use the real `PipeIO` when possible.

**Plan response:** integration tests use `MagicMock(spec=PipeIO)` to avoid booting bitbucket-pipes-toolkit's `Pipe(pipe_metadata=..., schema=...)` which requires a `pipe.yml` and schema dict. Phase 3 integration tests need to ASSERT on pipe.success / pipe.fail call arguments without depending on the toolkit's runtime semantics.

**Rationale:** isolates integration test scope to the HELM PATH, not the bitbucket-pipes-toolkit wrapper. Acceptable because:
- The `PipeIO` adapter is itself unit-tested elsewhere.
- Phase 3 integration tests focus on CHART-01/05 + HISTORY + META — orthogonal to pipe-io plumbing.

**Action item:** Plan-Checker can validate `MagicMock(spec=PipeIO)` is acceptable; if a stricter alternative is preferred, the test can construct a real `PipeIO()` and rely on `pipe.yml`'s minimal schema — that adds complexity without proportional test value.

### Deviation 5: Test 4 propagates HelmExecutionError out of `UpgradeAction.run` (NOT through cli.py's `pipe.fail`)

**ROADMAP Phase 3 SC2:** "an induced helm failure surfaces via `bitbucket-pipes-toolkit.fail()` with a non-zero exit code and human-readable message".

**Plan response:** Test 4 (`test_failure_path_surfaces_non_zero_exit_with_human_message`) calls `UpgradeAction.run` DIRECTLY and asserts via `pytest.raises(HelmExecutionError)`. cli.py's `except PipeError: pipe.fail(exc.user_message); return exc.exit_code` is tested by `tests/unit/test_cli.py::test_main_catches_pipe_error_from_upgrade_action` (Plan 03-04 Task 03-4-03).

**Rationale:** the integration test scope is "does helm failure produce a typed error?" — already proved by the exception assertion. cli.py's translation is covered at the unit-test level. Wrapping the integration test in a cli.main() invocation adds Subprocess scaffolding without testing anything new.

**Acceptable:** ROADMAP SC2 is satisfied collectively (unit + integration); the exact same failure SURFACES correctly via `pipe.fail` when invoked through cli.py — proven by the cli unit test.

**Action item:** if Plan-Checker insists on a cli.main()-driven integration test, add a 5th test that invokes `main(argv=[])` against a synthesized env (monkeypatch all required env vars) and asserts `main() != 0`.
