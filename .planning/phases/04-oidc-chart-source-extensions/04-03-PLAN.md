### Deviation 1: `os.environ.get("BITBUCKET_STEP_OIDC_TOKEN")` lives in `auth/__init__.py`, NOT in `Settings`

**Alternative considered:** add `bitbucket_step_oidc_token: str | None = Field(default=None, alias="BITBUCKET_STEP_OIDC_TOKEN")` to `Settings` in Plan 04-02.

**Plan response:** READ DIRECTLY from `os.environ` in `select_strategy`. Documented exception mirroring Phase 2's `_derive_session_name` reads of `BITBUCKET_PIPELINE_UUID` / `BITBUCKET_BUILD_NUMBER`.

**Rationale (per RESEARCH §7.2):**
1. Putting the token in `Settings` tempts callers to log the whole settings object — leaks the token.
2. Bypasses pydantic's masking (the field would have to be `SecretStr` AND every consumer would need to `.get_secret_value()` — surface area too wide).
3. The token carries no useful default beyond "present or absent".
4. The token is Bitbucket-platform-supplied (NOT consumer-supplied via pipe.yml) — same category as `BITBUCKET_PIPELINE_UUID`.

**Action item:** Plan-Checker should NOT flag the `os.environ.get` call in `auth/__init__.py` as an "outside settings.py" violation. The exception is documented here AND in the docstring of `select_strategy`.

(Inherited from Plan 04-02 Deviation 2 — same architectural choice. Errors raise at the composition root, not via pydantic validators.)

### Deviation 3: NO `Audience=` kwarg passed to `assume_role_with_web_identity`

**Naive read of CONTEXT D2:** "audience" might suggest it's passed to the STS API.

**Reality (per RESEARCH §7.1 + CONTEXT D2 explicit note):** `AssumeRoleWithWebIdentity` does NOT take an `Audience` arg. The `aud` claim lives INSIDE the JWT. **Plan response:** explicit negative test `test_get_credentials_does_not_pass_audience_kwarg_to_sts` proves no `Audience=` kwarg is ever sent. The unused `self._audience` field is stored but only consumed by potential debug logging (not in this plan; Phase 5+ might surface it).
