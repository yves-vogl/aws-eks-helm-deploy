### ADR-0001 — `0001-github-primary-forge.md`

- **Title:** GitHub is the primary forge; Bitbucket Pipe Marketplace stays as a compat target.
- **Context:** The pipe was developed historically on Bitbucket. v2 modernization (OIDC, cosign keyless, multi-arch native runners, release-please) requires features that are GitHub-native. RESEARCH Q5 + Phase 6 CONTEXT D1.
- **Decision Drivers:** OIDC keyless cosign requires GitHub Actions OIDC. Release-please-action is GitHub-native. Multi-arch ARM runners are GitHub-hosted. SLSA build provenance via `actions/attest-build-provenance`. GHCR is the native registry.
- **Considered Options:** (a) Stay Bitbucket-primary; (b) Move primary to GitHub, keep Bitbucket Pipe Marketplace listing; (c) Move primary to GitLab. Option (b) wins.
- **Decision Outcome:** GitHub primary forge; Bitbucket Pipe Marketplace listing maintained via minimal `bitbucket-pipelines.yml` (Phase 6 Plan 06-11). Docker Hub frozen at v1.3.0 (MIG-01).
- **Sources:** `.planning/phases/06-release-pipeline-supply-chain/06-CONTEXT.md`, REQUIREMENTS.md MIG-01.

### ADR-0002 — `0002-v2-clean-break.md`

- **Title:** v1.x → v2.x is a clean break; no compatibility shim layer.
- **Context:** v1.x is a shell pipe; v2 is a Python rewrite. Trying to maintain runtime compat would require dual code paths in every action.
- **Decision Drivers:** Maintenance overhead; security surface; clarity for consumers.
- **Considered Options:** (a) Compat shim layer; (b) Clean break with migration guide; (c) Versioned API.
- **Decision Outcome:** Clean break. v1.x frozen on Docker Hub at v1.3.0; v2.x publishes to GHCR. Migration guide (`docs/migration/v1-to-v2.md`) documents every breaking change.
- **Sources:** Phase 6 CONTEXT (clean-break decision), Phase 5 CONTEXT (META-02 default flip — biggest breaking change).

### ADR-0003 — `0003-cosign-keyless-over-gpg.md`

- **Title:** Cosign keyless (OIDC → Fulcio → Rekor) over GPG image signing.
- **Context:** Image signing for supply-chain provenance.
- **Decision Drivers:** GitHub Actions OIDC eliminates long-lived signing keys; Rekor transparency log; SLSA Build-Level-3; verifiable via `cosign verify` with cert constraints.
- **Considered Options:** (a) GPG sign; (b) Cosign keyless; (c) Cosign with stored key.
- **Decision Outcome:** Cosign keyless with `--bundle` for offline verify.
- **Sources:** `.planning/phases/06-release-pipeline-supply-chain/06-CONTEXT.md` D-cosign; REQUIREMENTS SEC-01.

### ADR-0004 — `0004-boto3-only-over-awscli.md`

- **Title:** `boto3`-only EKS token generation; drop bundled `awscli` from the runtime image.
- **Context:** v1 bundled `awscli` solely for `eks get-token`. ~120 MB image weight, slow cold-start.
- **Decision Drivers:** Image size; cold-start time (IMAGE-06); supply-chain narrowing.
- **Considered Options:** (a) Keep awscli; (b) boto3-only; (c) Static-link a Go binary.
- **Decision Outcome:** boto3-only. ~40 lines of Python in `src/aws_eks_helm_deploy/eks/token.py` generate `k8s-aws-v1.<base64url-presigned-STS-URL>`.
- **Sources:** `.planning/phases/02-aws-layer-auth-foundation/02-CONTEXT.md`; REQUIREMENTS AUTH-07.

### ADR-0005 — `0005-release-please-over-semversioner.md`

- **Title:** release-please-action over semversioner for release automation.
- **Context:** v1 used `semversioner` with a `.changes/` directory. v2 wants tighter Conventional-Commits + GitHub-native flow.
- **Decision Drivers:** Conventional Commits; release-PR model; eliminates `.changes/` ceremony; GitHub-native release creation.
- **Considered Options:** (a) Keep semversioner; (b) release-please; (c) Custom script.
- **Decision Outcome:** release-please-action v4 with `release-type: python`. Release-PR auto-updates pyproject.toml + CHANGELOG.md + pipe.yml image tag in one commit.
- **Sources:** `.planning/phases/06-release-pipeline-supply-chain/06-CONTEXT.md`; REQUIREMENTS CI-02.

### ADR-0006 — `0006-oidc-default-precedence.md`

- **Title:** When BOTH static keys AND OIDC token are present, static keys WIN (mirrors botocore default chain).
- **Context:** Phase 4 originally assumed OIDC precedence; Yves corrected the decision after consulting the AWS CLI behavior. Static keys win matches the boto3 default credential resolver chain — predictable for consumers.
- **Decision Drivers:** Principle of least surprise (matches AWS CLI); a one-time WARN log surfaces the precedence; consumers who accidentally configure both can see WHY their OIDC token was ignored.
- **Considered Options:** (a) OIDC wins (original Phase 4 spec); (b) Static wins (boto3 default); (c) Explicit env-var `AUTH_STRATEGY` to pick.
- **Decision Outcome:** Static keys win. WARN log `auth.precedence.static_keys_won_over_oidc` fires once per process.
- **Sources:** `.planning/phases/04-oidc-chart-source-extensions/04-CONTEXT.md` D1; REQUIREMENTS AUTH-04 (revised 2026-06-18).

### ADR-0007 — `0007-multi-arch-native-runners.md`

- **Title:** Multi-arch via native runners (`ubuntu-24.04` + `ubuntu-24.04-arm`); never QEMU.
- **Context:** Multi-arch image builds for `linux/amd64` + `linux/arm64`. QEMU silently produces broken arm64 images invisible to Bitbucket's amd64-only acceptance tests.
- **Decision Drivers:** Correctness (no silent broken builds); build time; native test execution on each arch.
- **Considered Options:** (a) QEMU on a single runner; (b) Two native runners with `docker buildx imagetools create` fan-in; (c) Ship amd64 only and let consumers cross-build.
- **Decision Outcome:** Two native runners; per-arch images pushed by digest; fan-in job assembles the multi-arch manifest.
- **Sources:** `.planning/phases/06-release-pipeline-supply-chain/06-CONTEXT.md` (multi-arch decision); REQUIREMENTS IMAGE-04.

### ADR-0008 — `0008-mkdocs-material-now-zensical-later.md`

- **Title:** mkdocs-material 9.x for v2.0; migrate to Zensical at v2.1+.
